# План автоматизации тестирования веб-сервиса "Путешествие дня"

## 1. Перечень автоматизируемых сценариев

### 1.1. Настройка тестового окружения (выполняется однократно перед запуском тестов)

1. Запустить контейнеры с MySQL, PostgreSQL и эмулятором банковских сервисов (gate-simulator) через Docker Compose.
2. Запустить целевое приложение (aqa-shop.jar) с подключением к MySQL.
3. Проверить доступность приложения по адресу http://localhost:8080.
4. Остановить приложение.
5. Запустить целевое приложение (aqa-shop.jar) с подключением к PostgreSQL.
6. Проверить доступность приложения по адресу http://localhost:8080.
7. Остановить приложение.

> **Примечание:** Все тестовые сценарии (разделы 1.2-1.5) выполняются дважды: сначала при запущенном приложении с MySQL,
> затем — с PostgreSQL.

### 1.2. Описание валидных тестовых данных

**Валидные данные карты:**

- **Номер карты:** 16 цифр, формат ХХХХ ХХХХ ХХХХ ХХХХ
- **Месяц:** две цифры 01-12, не ранее текущего месяца при текущем годе
- **Год:** две последние цифры года, не ранее текущего и не позднее текущего + 5 лет
- **Владелец:** имя и фамилия латиницей
- **CVC/CVV:** три цифры

**Тестовые карты эмулятора (из data.json):**

- `4444 4444 4444 4441` — статус APPROVED (одобрена)
- `4444 4444 4444 4442` — статус DECLINED (отклонена)

## 2. Перечень используемых инструментов

1. **IntelliJ IDEA** — среда разработки, в которой будет удобно писать, запускать и отлаживать автотесты.
2. **Java 11** — язык написания автотестов, так как целевое приложение написано на Java (JAR-файл), что обеспечит
   хорошую совместимость.
3. **Gradle** — система сборки и управления зависимостями. Более гибкая в настройке, чем Maven, имеет простой
   Groovy/Kotlin DSL. Позволит легко подключать плагины и управлять запуском тестов.
4. **JUnit Jupiter (JUnit5)** — фреймворк для написания и запуска повторяемых тестов. Самая актуальная версия JUnit.
5. **Selenide** — фреймворк для автоматизации UI-тестов. Удобен тем, что скрывает многие технические детали работы с
   Selenium WebDriver (управление браузером, ожидания), что ускоряет написание и повышает стабильность тестов.
6. **Docker / Docker Compose** — для запуска контейнеров с СУБД (MySQL, PostgreSQL) и эмулятора банковских сервисов (
   Gate Simulator). Это позволит легко разворачивать и переключать тестовое окружение, изолируя его от основной ОС.
7. **MySQL и PostgreSQL** — целевые СУБД. Тесты должны проверять работу приложения с обеими базами данных.
8. **Lombok** — библиотека для сокращения шаблонного кода (геттеры, сеттеры, конструкторы) в классах данных, что сделает
   код тестов чище.
9. **Java Faker** — библиотека для генерации тестовых данных (имена владельцев карт, случайные числа и т.д.). Поможет
   создавать реалистичные и разнообразные данные.
10. **Allure** — фреймворк для создания наглядных и информативных отчётов о тестировании. Позволит увидеть статистику
    прохождения тестов, прикрепить скриншоты к шагам и логи.
11. **Git / GitHub** — система контроля версий и репозиторий для хранения кода, тестов и отчетов. Обязательное
    требование диплома.

## 3. Возможные риски при автоматизации

1. **Отсутствие уникальных идентификаторов элементов на странице.**
    * *Проблема:* UI-элементы (кнопки, поля ввода) могут быть описаны без `test-id` или с нестабильными CSS-классами.
      Это сделает тесты хрупкими (будут ломаться при малейшем изменении верстки).
    * *Решение:* Использовать самые стабильные из доступных селекторов (в рамках диплома). В реальной работе - запросить
      у разработчиков добавление `data-test-id` атрибутов.

2. **Сложности с эмулятором банковских сервисов.**
    * *Проблема:* Эмулятор может работать нестабильно или его логика может не полностью соответствовать реальным
      сервисам.
    * *Решение:* Внимательно изучить документацию и `data.json`. Добавить проверки на то, что эмулятор запущен и
      отвечает перед стартом тестов.

3. **Большое количество багов в приложении.**
    * *Проблема:* В задании сказано, что приложение "усыпано багами". Это значит, что многие тесты упадут. Важно
      отличать падение теста из-за бага от падения из-за ошибки в самом тесте.
    * *Решение:* Тщательно анализировать каждый упавший тест и заводить баг-репорты в Issues на GitHub.
