   Запущены контейнеры с MySQL, PostgreSQL и эмулятором банковских сервисов (gate-simulator на порту 9999).

1.2. Источники тестовых данных:
Валидные номера карт (из файла gate-simulator/data.json):

4444 4444 4444 4441 — карта со статусом APPROVED (одобряется банком).

4444 4444 4444 4442 — карта со статусом DECLINED (отклоняется банком).

Невалидные номера карт:

Карта с номером, отсутствующим в базе эмулятора (например, 1234 5678 9012 3456).
Карта с неверным количеством цифр (15 или 17 цифр).
Карта с буквами или спецсимволами в номере.

2. Перечень используемых инструментов

1. IntelliJ IDEA — среда разработки, в которой будет удобно писать, запускать и отлаживать автотесты.

2. Java 11 — язык написания автотестов, так как целевое приложение написано на Java (JAR-файл), что обеспечит хорошую
   совместимость.

3. Gradle — система сборки и управления зависимостями. Более гибкая в настройке, чем Maven, и имеет простой
   Groovy/Kotlin DSL. Позволит легко подключать плагины и управлять запуском тестов.

4. JUnit Jupiter (JUnit5) — фреймворк для написания и запуска повторяемых тестов. Самая актуальная версия JUnit.

5. Selenide — фреймворк для автоматизации UI-тестов. Удобен тем, что скрывает многие технические детали работы с
   Selenium
   WebDriver (управление браузером, ожидания элементов), что ускоряет написание и повышает стабильность тестов.

6. Docker / Docker Compose — для запуска контейнеров с СУБД (MySQL, PostgreSQL) и эмулятора банковских сервисов (Gate
   Simulator). Это позволит легко разворачивать и переключать тестовое окружение, изолируя его от основной ОС.

7. MySQL и PostgreSQL — целевые СУБД. Тесты должны проверять работу приложения с обеими базами данных.

8. Lombok — библиотека для сокращения шаблонного кода (геттеры, сеттеры, конструкторы) в классах данных, что сделает код
   тестов чище.

9. Java Faker — библиотека для генерации тестовых данных (имена владельцев карт, случайные числа и т.д.). Поможет
   создавать реалистичные и разнообразные данные.

10. Allure — фреймворк для создания наглядных и информативных отчётов о тестировании. Позволит увидеть статистику
    прохождения тестов, прикрепить скриншоты к шагам и логи.

11. Git / GitHub — система контроля версий и репозиторий для хранения кода, тестов и отчетов. Обязательное требование
    диплома.


3. Возможные риски при автоматизации

1. Отсутствие уникальных идентификаторов элементов на странице.
   UI-элементы (кнопки, поля ввода) могут быть описаны без test-id или с нестабильными CSS-классами. Это сделает
   тесты хрупкими (будут ломаться при малейшем изменении верстки).

   Решение: использовать самые стабильные из тех что есть (в рамках диплома, в реальной работе - попробовать попросить
   разработчиков добавить data-test-id)

2. Сложности с эмулятором банковских сервисов.
   Эмулятор может работать нестабильно или его логика может не полностью соответствовать реальным сервисам.

Решение: Внимательно изучить документацию и data.json. Добавить проверки на то, что эмулятор запущен и отвечает, перед
стартом тестов. (если будет нужно)

3. Большое количество багов в приложении.

В задании сказано, что приложение "усыпано багами". Это значит, что многие тесты упадут. Важно отличать
падение теста из-за бага от падения из-за ошибки в самом тесте.

Решение: Тщательно анализировать каждый упавший тест и заводить баг-репорты в Issues на GitHub.
